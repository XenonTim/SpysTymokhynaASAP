# ASAP Messenger — об'єднаний проєкт

Real-time месенджер за ТЗ "Real-time Месенджер «ASAP»" (Спис К.Д., Тимохина К.В.).

## Що тут об'єднано

У вихідному репозиторії було два окремих, ще не зістикованих шари:

1. **`asap/`** — повноцінне ядро застосунку: TCP-сервер з кастомним бінарним
   протоколом (`Packet` / `PacketType` / `PayloadBuilder`), багатопотоковість
   (thread pool на з'єднання), MongoDB-репозиторії, bcrypt для паролів,
   AES-утиліта, ролі User/Admin, консольний `TestClient`.
2. **Корінь репо (`src/`)** — нарис веб-інтерфейсу (Servlet + JSP + статичний
   HTML-макет чату), ще не підключений до сервера: своя модель `User`, своє
   підключення до MongoDB Atlas, форми логіну/реєстрації без жодної реальної
   логіки чату.

Ці два шари використовували різні стеки (сирі TCP-сокети проти Servlet/JSP
поверх Tomcat), тому "просто склеїти" файли було неможливо технічно: HTTP-сервер
не вміє віддати браузеру сирий TCP-socket. Замість Tomcat/Servlet я додала
легкий **WebGateway** (пакет `org.example.gateway`, без зовнішніх залежностей) —
єдиний новий шар, який:

- роздає браузеру статичні сторінки клієнта (`webapp/login.html`,
  `register.html`, `chat.html`);
- приймає WebSocket-з'єднання на `/ws` і для кожного з них відкриває окреме
  TCP-з'єднання до основного `Server` (порт 8080), перекладаючи JSON
  (зручний для JS) у ті самі `Packet`/`PacketType`, якими вже оперує сервер.

Сам протокол, сервер, шифрування паролів, БД-логіка і багатопотоковість
**не змінювалися** — гейтвей лише транслює повідомлення, як ще один клієнт
(аналогічно `TestClient`, тільки керований через WebSocket замість консолі).

## Структура фінального проєкту

```
ASAP-Messenger/
├── pom.xml
├── webapp/                      # статичний веб-клієнт (без збірки/бандлера)
│   ├── login.html
│   ├── register.html
│   ├── chat.html
│   ├── style.css
│   └── asap-client.js
└── src/
    ├── main/java/org/example/
    │   ├── server/network/      # Server, ClientHandler, SessionRegistry — без змін
    │   ├── client/               # TestClient + мережевий шар клієнта — без змін
    │   ├── db/                   # репозиторії + MongoDBConnection (додано підтримку Atlas)
    │   ├── models/                # User, Chat, Message — без змін
    │   ├── shared/protocol/       # Packet, PacketIO, PacketType, PayloadBuilder — без змін
    │   ├── shared/security/       # AesUtil, PasswordUtil — без змін
    │   └── gateway/               # НОВЕ: WebGateway, WebSocketConnection, ClientSessionBridge, SimpleJson
    └── test/java/org/example/    # існуючі тести + SimpleJsonTest
```

## Налаштування MongoDB Atlas

`MongoDBConnection` тепер бере рядок підключення з системної властивості або
змінної середовища `MONGO_URI` (і опційно `MONGO_DB_NAME`), з фолбеком на
локальний `mongodb://localhost:27017`.

1. Створіть кластер на [MongoDB Atlas](https://www.mongodb.com/atlas) (є
   безкоштовний тариф M0) і користувача БД з паролем.
2. У Network Access додайте свою IP-адресу (або `0.0.0.0/0` для тестування).
3. Скопіюйте рядок підключення (Drivers → Java), наприклад:
   ```
   mongodb+srv://<user>:<password>@<cluster>.mongodb.net/?retryWrites=true&w=majority
   ```
4. Перед запуском кожного компонента (сервера, AdminSeeder) задайте змінну
   середовища:

   **Linux/macOS:**
   ```bash
   export MONGO_URI="mongodb+srv://<user>:<password>@<cluster>.mongodb.net/?retryWrites=true&w=majority"
   ```

   **Windows (PowerShell):**
   ```powershell
   $env:MONGO_URI="mongodb+srv://<user>:<password>@<cluster>.mongodb.net/?retryWrites=true&w=majority"
   ```

   Або передайте напряму як `-D` параметр Maven (див. нижче).

## Запуск

Усі команди виконуються з кореня `ASAP-Messenger/` (там, де `pom.xml`).

### 0. Збірка

```bash
mvn clean install
```

Має пройти `mvn test` (юніт-тести AesUtil, PasswordUtil, PayloadBuilder,
SessionRegistry, SimpleJson) і зібрати `target/SpysTymokhynaASAP-1.0-SNAPSHOT.jar`.

### 1. (Один раз) Створити акаунт адміністратора

```bash
mvn exec:java -Dexec.mainClass="org.example.db.AdminSeeder" -DMONGO_URI="mongodb+srv://..."
```

Виведе логін/пароль адміна (`admin` / `Admin1234!`) — запишіть, знадобиться
для перевірки адмін-функцій.

### 2. Запустити основний сервер (термінал №1)

```bash
mvn exec:java -Dexec.mainClass="org.example.server.network.Server" -DMONGO_URI="mongodb+srv://..."
```

Побачите: `[Server] ASAP Messenger started on port 8080`.

### 3. Запустити веб-гейтвей (термінал №2)

```bash
mvn exec:java -Dexec.mainClass="org.example.gateway.WebGateway"
```

Побачите: `[WebGateway] Запущено на http://localhost:8081`.

> Важливо: запускайте цю команду саме з кореня проєкту — гейтвей шукає
> каталог `webapp/` відносно поточної робочої директорії.

### 4. Відкрити веб-клієнт

Перейдіть у браузері на **http://localhost:8081/login.html**.

### 5. (Опційно) Консольний клієнт для діагностики протоколу

```bash
mvn exec:java -Dexec.mainClass="org.example.client.TestClient"
```

Дає прямий доступ до команд `register`, `login`, `msg`, `history`, `ban`,
`stats` і зручний для демонстрації самого протоколу викладачу окремо від
браузера.

## Як перевірити дієздатність (сценарій для здачі)

Все нижче можна робити з двох вкладок браузера (звичайний режим + інкогніто,
щоб не плутались сесії в `sessionStorage`).

1. **Реєстрація** — вкладка 1: відкрийте `/register.html`, зареєструйте
   `kate` / `pass1234`. Має автоматично залогінити й перенести на `/chat.html`.
2. **Друга реєстрація** — вкладка 2 (інкогніто): зареєструйте `kseniia` /
   `pass1234`.
3. **Приватний чат** — у вкладці 1 введіть логін `kseniia` в полі "Логін
   користувача" зліва, тип "Приватний", натисніть `+`. Чат з'явиться у списку
   й одразу відкриється.
4. **Real-time доставка** — у вкладці 2 той самий чат теж з'явиться в списку
   (можливо, після GET_CONTACTS — оновіть сторінку, якщо список ще порожній,
   або зачекайте на CHAT_CREATED, якщо чат створювала ця ж вкладка). Напишіть
   повідомлення у вкладці 1 — воно має миттєво з'явитись у вкладці 2 без
   перезавантаження сторінки.
5. **Історія чату** — перезавантажте вкладку 2 і знову відкрийте той самий
   чат: попередні повідомлення мають підвантажитись з MongoDB
   (`GET_HISTORY` → `HISTORY_RESPONSE`).
6. **Індикація онлайн/офлайн** — закрийте вкладку 2: на сервері в консолі
   побачите `[Server] Client disconnected: kseniia`.
7. **Груповий чат** — створіть чат типу "Груповий" з третім користувачем
   (зареєструйте його в ще одній вкладці) — перевірте, що повідомлення
   доходять до всіх учасників.
8. **Адмін-функції** — увійдіть як `admin` / `Admin1234!` в окремій вкладці.
   У лівій панелі з'явиться блок "Адмін-панель":
   - Бан: введіть `kate`, натисніть "Бан" — у вкладці з `kate` (якщо вона
     була онлайн) має статись миттєвий розрив з'єднання і повернення на
     `/login.html` з повідомленням про блокування. Повторний логін під
     `kate`/`pass1234` має повертати помилку "account is banned".
   - Статистика: кнопка "Статистика серверу" покаже кількість активних
     з'єднань (тост у нижній частині екрана).
9. **Перевірка тестів і протоколу окремо від UI** — запустіть `TestClient` і
   повторіть команди `register`/`login`/`msg`/`ban`/`stats` вручну: це
   показує, що кастомний протокол працює незалежно від веб-шару.

## Відомі обмеження, варті уваги перед захистом

- **AES-шифрування повідомлень підготовлене, але не підключене.**
  У `ClientHandler` генерується сесійний `SecretKey` (`AesUtil.generateKey()`),
  але жодного handshake обміну ключем із клієнтом і жодного реального
  `AesUtil.encrypt/decrypt` викликів на вміст повідомлення в коді `asap/`
  немає — `content` зберігається й пересилається як є. Це вже існувало у
  вихідній реалізації, я це не змінювала. Я додала легкий XOR+Base64 шар на
  стороні браузера (`chat.html`) як демонстрацію шифрування "в дорозі" для
  вимоги ТЗ п.6, але це не замінює повноцінний AES — якщо потрібно закрити
  цю вимогу повністю, варто або реалізувати handshake обміну AES-ключа
  клієнт↔сервер, або хоча б показати викладачу `AesUtilTest` як доказ, що
  механізм шифрування існує і протестований на рівні utility-класу.
- **`CREATE_CHAT` не валідує вхідні дані** (наприклад, порожній `members`
  спричинить виняток на сервері) — для навчального проєкту прийнятно, але
  плутана помилка прийде як `ERROR` без деталей.
- **WebGateway тримає по одному TCP-з'єднанню на кожну відкриту вкладку
  браузера** (так само як було б з кількома консольними `TestClient`) —
  це узгоджується з вимогою "сервер створює окремий потік для кожного
  підключення клієнта" і не є помилкою, просто варто розуміти при підрахунку
  активних з'єднань у "Статистиці серверу".
